Kanban

KANBAN po japońsku znaczy spis widoczny

Metoda ta pozwala na organizację pracy. Opiera się na podziale na 3 kolumny czasami na więcej, a te kolumny to:

  • TODO = do zrobienia
  • DOING = w trakcie
  • DONE = zrobione

Może być jeszcze więcej kolumn:

  • Testing - testowanie
  • inReview - do przejrzenia

Kanban - podsumowanie metody:

  • Ułatwia śledzenie postępu prac / nauki
  • Zalecane trzymanie jednego zadania w "doing"
  • Dobrze opisuj "definition of done"
  • Można stosować offline
  • Online za darmo możesz przygotować swoją tablicę na trello.com

Jak działa internet i przeglądarka?

Przeglądarki korzystają z protokołu HTTP (w większości przypadków). Dzięki niemu mogą pobrać wszystkie pliki i je wyświetlić na ekranie monitora.

HTTP to w rozwinięciu Hypertext Transfer Protocol czyli jest to protokół do transportowania html'a i całej reszty.

Jest też coś takiego jak HTTPS czyli Hypertext Transfer Protocol Secure i jest on bezpieczniejszy, ponieważ jest szyfrowanym protokołem. TO jest "połączenie z kłódką". Jest rekomendowane żeby wszędzie korzystać z tego protokołu. Zasada działania jest praktycznie taka sama jak HTML.

URL czyli Uniform Resoure Locator czyli adres, który wpisujemy.

Co się dzieje w momencie wpisywania adresu strony w przeglądarce?

  1. wpisuję w pasku adresu cotenfronted.pl [enter]
  2. przeglądarka szuka serwera cotenfronted.pl [DNS]
  3. po znalezieniu serwera odpytuje go o index (jeśli podamy sam adres serwera) - to co jest indexem zależy od konfiguracji serwera (nie musi to by c index.html, choć w przypadku stron statycznych jest to najczęstszy przypadek)
  4. serwer wysyła do przeglądarki zawartość html
  5. przeglądarka wczytuje html i szuka wszystkich dodatkowych zasobów: style, skrypty, obrazki itp. i dla każdego z zasobów wykonuje zapytanie
  6. przeglądarka parsuje i renderuje stronę

Komunikacja między serwerem, a przeglądarką jest w modelu request - response (zapytanie - odpowiedź). Czyli przeglądarka o coś pyta serwer, a ten wysyła jej informację zwrotną.

Narzędzia deweloperskie w Chrome'ie dają nam możliwość podejrzenia jak funkcjonuje komunikacja między przeglądarką, a serwerem. Możemy to podejrzeć w zakładce network. Widać tam co się w jakim momencie ładowało w przeglądarce. 🙂

Kody odpowiedzi HTTP:

  • 200 - OK, gdy dany zasób znajduje się na serwerze
  • 30x - szukasz w złym miejscu - przekieruję Cię w dobre (bo wiem gdzie)
  • 404 - nie mam tego czego potrzebujesz. Sorry
  • 50x - Nie czuję się najlepiej. Błąd wewnętrzny serwera.

Czym jest root (/)?

"/" mówi nam, którym miejscu przeglądarka musi szukać zasobu. Ma on znaczenie przy odwołaniu do ścieżki względnej i niewzględnej. Czyli w odwołaniu do obrazka przed lokalizacją obrazka użyjemy / (np. src="/img/wtf.svg") to on się zawsze otworzy, Bez względu na to czy będziemy wchodzić przez stronę główną lub podstronę. Jest to odwołanie bezwzględna.

Jeśli przed lokalizacją obrazka nie użyjemy "/" (np. src="img/wtf.svg") to obrazek otworzy się tylko wchodząc przez stronę główną. Dzieje się tak ponieważ w drugim przypadku ścieżka jest dopisywana do podstrony i szuka wtedy przeglądarka pliku w złym miejscu.

Tworząc stronę uruchamianą "z dysku" pomijamy używanie slasha z przodu.

Posumowanie:

  • protokół który obsługuje strony internetowe i ich zawartość to HTTP (aktualnie coraz częściej HTTP/2)
  • komunikacja przeglądarka - serwer odbywa się na zasadach request - response
  • odpowiedzi serwera mają różne kody (i pozwala nam to na diagnozowanie problemów)
  • root (/) jest istotną częścią ścieżek do zasobów (i dobrze wiedzieć jak działa 🙂 )

WTF: Terminal

Każdy z systemów operacyjnych posiada swój terminal. MacOs i linux mają terminale bardzo podobne, a Windows ma swój poleceń, który trochę odstaje od terminali z innych systemów. W szkoleniu WTF: Co ten frontend poznajemy obsługę wszystkich tych terminali, z różnych systemów.

Przed przystąpieniem do pracy w terminalu trzeba skonfigurować VSC, tak aby mógł dobrze współpracować z terminalem. Pomocna w tym będzie poniższa komend, którą trzeba będzie wpisać po kliknięciu ctrl+shift+p:

Shell Command: Install 'code' command in PATH
Polecenia terminala w systemie Windows

Żeby odpalić terminal na Windzie wchodzimy w menu Start i zaczynamy wpisywać cmd. Otworzy się nam wiersz poleceń (Windowsowy odpowiednik terminala) i możemy już kontynuować pracę z terminale używając poniższych komend.

  • cd C:\ - przejście do głównego katalogu na dysku C,
  • dir - wypisanie zawartości katalogu.
  • cd .. - przejście do katalogu wyżej,
  • cd nazwa-katalogu - przejście do katalogu o nazwie nazwa-katalogu,
  • code . - uruchomienie visual studio code w aktualnym katalogu,
  • cd - wyświetlenie aktualnej ścieżki,
  • mkdir nazwa-katalogu - stworzenie katalogu o nazwie nazwa-katalogu
Polecenia terminala w systemie MacOs

Żeby odpalić terminal an Mac'u wciskamy kombinację klawiszy command+spacja. Otworzy się okno o nazwie spotlight search i wpisujemy: terminal. Otworzy się nam terminal i możemy już kontynuować pracę z terminale używając poniższych komend.

  • cd ~ - przejście do głównego katalogu na dysku C,
  • ls - wypisanie zawartości katalogu.
  • cd .. - przejście do katalogu wyżej,
  • cd nazwa-katalogu - przejście do katalogu o nazwie nazwa-katalogu,
  • code . - uruchomienie visual studio code w aktualnym katalogu,
  • pwd - wyświetlenie aktualnej ścieżki,
  • mkdir nazwa-katalogu - stworzenie katalogu o nazwie nazwa-katalogu
Polecenia terminala w systemie Linux

Żeby odpalić terminal na Linux'ie wybieramy Terminal w Odpalaczu aplikacji. Otworzy się nam terminal i możemy już kontynuować pracę z terminale używając poniższych komend.

Jest GIT!

GIT to system kontroli wersji. Ułatwia on zarządzanie projektem. Pozwala on na łatwe:

  • powrócenie do sprawnej wersji, gdy wkradnie nam się błąd do kodu i sprawi, że program przestanie działać.
  • Umożliwia również na wrócenie do poprzedniej wersji, gdy okaże się, że była lepsza niż obecna po wprowadzonych zmianach,
  • lub po przypadkowym usunięciu jakiegoś pliku.
  • Umożliwia on wprowadzenie zmian w kilku plikach (np. w celu zademonstrowaniu kilku opcji klientowi), a później na łatwe wprowadzenie cofnięcie wprowadzonych zmian.
  • umożliwia na pracę nad projektem grupowym, podzielonym na kilka segmentów, nad którymi pracują różne osoby. Łatwa jest wymiana plikami pomiędzy członkami zespołu i kontrola postępu pracy (podgląd wprowadzonych zmian).

Program GIT umożliwia nam śledzenie postępu prac. Czyli możemy krok po kroku zobaczyć jak powstawał projekt i jakie zmiany były w nim wprowadzone i w jakiej kolejności.

Instalujemy GIT'a

https://git-scm.com

Z powyższego linku ściągamy GIT'a na swój komputer i go instalujemy. Program obsługujemy poprzez komendy wpisywane w terminalu. Żeby sprawdzić czy GIT się zainstalował wpiszemy swoją pierwszą komendę:

git --version

Powinna pokazać się informacja o wersji zainstalowanej na naszym komputerze:

git version 2.28.0.windows.1

Następnym krokiem jest konfiguracja Gita poprzez wpisanie dwóch komend: 

                        
                            git config --global user.name "Jan Kowalski"
                            git config --global user.email jankowalski@example.com

Informacje te służą temu, żeby podać kim jesteśmy. Jeśli gdzieś się pomylimy w trakcie wpisywania danych, możemy je poprawić poprzez ponowne wpisanie komendy już poprawionymi danymi.

Żeby sprawdzić czy dane prawidłowo się wprowadziły, wpisujemy komendę: 

                        
                            git config --get user.name
                            git config --get user.email

Powinniśmy uzyskać odpowiedź:

                        
                            Jan Kowalski
                            jankowalski@example.com

Konfigurację konta robimy tylko raz, na samym początku korzystania konta użytkownika na danym komputerze.

Git składa się z repozytoriów (projektu) czyli wszystkie pliki jakie wchodzą skład danego projektu (katalogu).

Zakładamy pierwsze repozytorium

Zakładamy je w katalogu homepage do tego celu wykonujemy komendę:

git init

Pojawi się poniższy komunikat:

Initialized empty Git repository in C:/dev/homepage/.git/

Oznacza to, że utworzone zostało puste repozytorium homepage w katalogu homepage/.git/ Katalog ten jest ukryty, którego nie możemy skasować (nie powinniśmy, ponieważ zawiera ważne informacje). W tym katalogu znajduje się cała historia programy git.

Jak często commit'ować?

Na początku warto często commitować, żeby wyrobić sobie nawyk. Watro robić commit'a każdej większej zmianie (np. dodanie sekcji w artykule, dodanie stylów do jakiejś klasy).

Polecienia cd.

gitk
pokazuje nam:
  • całą historię commit'ów,
  • pokazuje nasze dane,
  • autorów poszczególnych commit'ów,
  • widzimy tytuły commit'ów,
  • widzimy, które pliki się zmieniły (możemy podglądać pliki i obserwować co się w nich zmieniło),
  • datę kiedy dany commit został zrobiny,
clear
polecenie terminala, które czyści nam okno danego terminala

W Visual Studio Code jest narzędzie: Git (jest ono wbudowane w VSC), które pozwala nam na podglądanie repozytorium. Dodatkowo możemy sobie zainstalować rozszerzenie GitLens. jest to już dość mocno rozbudowane narzędzie, które daje większe możliwości kontroli zmian wprowadzonych w plikach.

GitHub

Jest to narzędzie, które pozwoli na pokazywanie światu naszego kodu bez konieczności wklejania go na CodePen'a lub inne tego typu serwisy. Pozwala on an trzymanie naszych repozytoriów poza naszym komputerem. Na początek:

  1. Zakładamy konto na GitHub'ie: https://github.com - podajemy prawdziwe dane, najlepiej swoje imię i nazwisko. Jest to nasza programistyczna wizytówka, dlatego dbamy o to żeby była jak najlepsza. Często w rozmowie o pracę podajemy link do serwisu, gdzie pracodawca może zobaczyć jak pracujemy.
  2. Zakładamy repozytorium w serwisie GitHub.
  3. Podpinamy repozytorium zdalne do lokalnego. Wykonujemy to za pomocą dwóch linijek kodu wpisanych w terminalu, które dostaniemy po utworzeniu repozytorium na GitHubie:
                                    
                                    git remote add origin https://github.com/imię-nazwisko/homepage.git
                                    git push -u origin master
    Jeśli nie mam na dysku plików utworzonego repozytorium, to możemy je stworzyć na dysku po wykorzystaniu poniższego kodu: 
                                    
                                    echo "# homepage1" >> README.md
                                    git init
                                    git add README.md
                                    git commit -m "first commit"
                                    git remote add origin https://github.com/imię-nazwisko/homepage.git
                                    git push -u origin master
  4. Gdy już po powiązaniu repozytorium lokalnego z zdalnym i dalej pracujemy z kodem (wprowadzamy jakieś zmiany w projekcie i je commit'ujemy), to kolejne wysłanie danych o repozytorium na GitHub'a będzie możliwe za pomocą komendy:
    git push
  5. Kroki przy commit'owaniu:
    1. git status
    2. git add .
    3. git commit -m "nazwa commit'a"
    4. git status
    5. git push

Podsumowanie

  • GIT pozwala na sprawne zarządzanie zmianami w projekcie.
  • Działa lokalnie (GIT).
  • Do działania zdalnego potrzebuje serwera (np. GitHub).
  • Znajomość podstaw GIT-a jest wymagana niezależnie od technologii.
  • Konto na GitHubie jest Twoją programistyczną wizytówką, zadbaj aby była ładna i prawdziwa 🙂
  • Commitujemy jak dokładamy jakąś widoczną zmianę do projektu (lepiej częściej, niż za mało)
  • Podstawowy flow:
    • git add .
    • git commit -m "opis zmiany"
    • git push
